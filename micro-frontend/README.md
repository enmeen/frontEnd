# 微前端学习笔记

## 什么是微前端？

微前端是一种架构风格，将单页面应用拆分为多个独立的小型前端应用，这些应用可以独立开发、独立部署，最终组合成一个完整的应用。

## 核心价值

### 1. 独立开发
- 不同团队可以并行开发
- 技术栈可选，团队可以使用熟悉的技术
- 降低代码冲突风险

### 2. 独立部署
- 子应用可以独立发布，不影响其他应用
- 支持灰度发布和A/B测试
- 降低部署风险

### 3. 渐进式升级
- 可以逐步重构老旧系统
- 新功能可以用新技术栈开发
- 系统现代化改造的平滑路径

## 基本架构模式

### 主从模式 (Main-Sub)
```
Main App (主应用)
├── Sub App 1 (子应用1)
├── Sub App 2 (子应用2)
└── Sub App 3 (子应用3)
```

### 路由分发模式
- 基于路由加载不同的子应用
- 子应用间相互独立
- 适合业务场景清晰的应用

### 应用容器模式
- 主应用作为容器
- 子应用作为组件嵌入
- 适合组件化的业务场景

## 应用场景

### 适合微前端的场景
- 大型企业应用，多团队协作
- 系统需要渐进式重构
- 不同业务模块技术栈差异大
- 需要独立发布和部署

### 不适合的场景
- 小型项目，团队单一
- 应用间耦合度高
- 性能要求极致的单页应用
- 简单的业务逻辑

## 核心挑战

1. **应用加载与管理** - 如何动态加载和卸载子应用
2. **路由管理** - 如何协调主应用和子应用的路由
3. **状态共享** - 如何在应用间共享数据和状态
4. **样式隔离** - 避免CSS样式相互影响
5. **通信机制** - 应用间如何进行数据通信
6. **性能优化** - 避免重复加载和资源浪费

## 学习路径

1. **理解基础概念** - 什么是微前端，为什么需要
2. **学习主流方案** - Module Federation、Qiankun、Single-SPA等
3. **掌握核心技术** - 应用加载、路由、通信等
4. **实践对比** - 不同方案的优缺点和适用场景
5. **选择方案** - 根据实际需求选择合适的方案
